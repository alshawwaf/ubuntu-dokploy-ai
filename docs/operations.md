## Network topology

The agentic stack (n8n, Ollama, Open WebUI, etc.) runs under a Dokploy compose project named `cp-agentic-mcp-playground-<suffix>` — the suffix is generated by Dokploy; find yours with `docker ps | grep open-webui`. Its internal Docker network follows the same pattern: `cp-agentic-mcp-playground-<suffix>_demo`.

Standalone Dokploy apps (AI Guardrails Demo, Dev Hub, etc.) each get their **own isolated network** and can't resolve container names from the agentic stack's private `demo` network. To bridge Ollama specifically, `ollama-cpu` is attached to the shared **`dokploy-network`** (in the playground compose), so any Dokploy app on that network reaches it by name at **`http://ollama-cpu:11434`** — no host IP, no published port, survives redeploys and IP changes. AI Guardrails's `.env` uses exactly that URL. (Ollama is intentionally **not** published to the host, so the old `<SERVER_IP>:11434` hairpin does not work and should not be used.)

## Docker address pools & the 192.168 SSH-lockout trap

**Symptom.** After many deploys, SSH from a `192.168.x` management LAN suddenly stops working — yet the VM console works, `sshd` is `active (running)` and even accepting logins from other hosts, the disk is fine, and disabling `ufw` makes no difference. A reboot doesn't fix it.

**Cause.** Docker's *built-in* address pools hand out `172.17`–`172.31.0.0/16` (15 networks) and then **spill into `192.168.0.0/16`**. Once the stack has ~16 compose networks, the next one lands on `192.168.0.0/20` — which **contains** a typical admin LAN such as `192.168.5.0/24`. The host then routes replies to your SSH client into a dead Docker bridge instead of back out the WAN, so the connection times out. It survives reboots because Docker recreates the bridge on boot.

**Prevention (built in).** The installer now pins Docker's user-network pools to `10.201/10.202.0.0/16` (`/24` size = 512 networks) in `/etc/docker/daemon.json` *before* any network is created, so app networks never touch `192.168.x`. Confirm:

```bash
cat /etc/docker/daemon.json
```

**Diagnose (from the console if you're locked out):**

```bash
ip route | grep -F 192.168
sudo docker network inspect $(sudo docker network ls -q) -f '{{.Name}} {{range .IPAM.Config}}{{.Subnet}}{{end}}' | grep -F 192.168
```

A `192.168.x.x/NN dev br-...` route plus a Docker network on a `192.168` subnet is the culprit.

**Recover:**

```bash
sudo ip route del 192.168.0.0/20
```

(Use the exact `192.168.x.x/NN` from `ip route`.) SSH returns immediately — replies fall back to the default gateway. Then make it durable so it can't recur:

```bash
sudo tee /etc/docker/daemon.json >/dev/null <<'JSON'
{
  "default-address-pools": [
    { "base": "10.201.0.0/16", "size": 24 },
    { "base": "10.202.0.0/16", "size": 24 }
  ]
}
JSON
sudo systemctl restart docker
sudo docker ps -a --filter network=<offending_net> -q | xargs -r sudo docker rm -f
sudo docker network rm <offending_net>
sudo docker network prune -f
```

Then redeploy the affected stack (or re-run the installer) so it recreates its network from the `10.x` pool.

## Ollama model management

Which models are pulled (and set as the Open WebUI default) is controlled by the agentic env file, not hardcoded. See `OLLAMA_MODELS` and `OPEN_WEBUI_DEFAULT_MODELS` in [`automation/envs/.env_agentic.example`](../automation/envs/.env_agentic.example). Shipped default:

```
OLLAMA_MODELS=gemma4:e2b,qwen3.5:9b,richardyoung/mythos-9b-unhinged-abliterated:latest,huihui_ai/deepseek-r1-abliterated:8b,qwen3-embedding:0.6b
OPEN_WEBUI_DEFAULT_MODELS=gemma4:e2b
OLLAMA_MAX_LOADED_MODELS=2
```

The pull is performed by the `ollama-pull-models-cpu` (or `gpu`) service in the [`cp-agentic-mcp-playground`](https://github.com/alshawwaf/cp-agentic-mcp-playground) compose. Set the variables to a comma-separated list to pull more. First-time pulls run to completion automatically and can take 10–20 minutes depending on model size.

```bash
docker exec ollama-cpu ollama list          # list models
docker exec ollama-cpu ollama pull <model>   # pull a model
docker exec ollama-cpu ollama rm <model>     # remove a model
```

> The compose uses `pull_policy: always` for Ollama (needed for new model architecture support) and `pull_policy: if_not_present` for Open WebUI (avoids startup failures from `ghcr.io` registry issues or broken `:main` builds).

## Ollama model errors

**"500: unable to load model" with a blob path** — the model file is corrupt or partially downloaded. Remove and re-pull:

```bash
docker exec ollama-cpu ollama rm <model>
docker exec ollama-cpu ollama pull <model>
```

If pulls fail, check disk with `df -h` — each model needs several GB.

**"500: unable to load model" right after pulling** — the Ollama version may be too old for the model architecture. Update Ollama, then recreate the container (see [project name](#docker-compose-project-name) for the `-p` flag):

```bash
docker pull ollama/ollama:latest
cd /etc/dokploy/compose/<project-dir>/code
docker compose -p <project-name> --profile cpu up -d --force-recreate ollama-cpu
```

## Docker Compose project name

Always pass `-p cp-agentic-mcp-playground-<suffix>` when running compose commands against the agentic stack. Without it, Docker uses the directory name (`code`) as the project, creating orphan networks and breaking container DNS.

```bash
cd /etc/dokploy/compose/cp-agentic-mcp-playground-<suffix>/code
docker compose -p cp-agentic-mcp-playground-<suffix> --profile cpu up -d --no-deps open-webui
```

## Traefik domain labels (404s)

The agentic compose uses `{{DOMAIN}}` placeholders in Traefik labels, substituted at deploy time by the automation. If a router rule is literally `Host('chat.{{DOMAIN}}')` and Traefik returns 404, the compose was recreated from the raw file (e.g. after a `git pull`) without substitution:

```bash
sed -i 's/{{DOMAIN}}/<YOUR_DOMAIN>/g' \
  /etc/dokploy/compose/cp-agentic-mcp-playground-<suffix>/code/docker-compose.yml
cd /etc/dokploy/compose/cp-agentic-mcp-playground-<suffix>/code
docker stop open-webui && docker rm open-webui
docker compose -p cp-agentic-mcp-playground-<suffix> --profile cpu up -d --no-deps open-webui
```

## Open WebUI can't reach Ollama

If Open WebUI was recreated without the correct project name, it lands on the wrong network (`code_demo` instead of `cp-agentic-mcp-playground-<suffix>_demo`). Recreate with the correct `-p` (above). Verify:

```bash
docker inspect open-webui --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool | grep -E '"cp-agentic|IPAddress'
# should show cp-agentic-mcp-playground-<suffix>_demo
```

## AI Guardrails settings database

AI Guardrails stores runtime settings (API keys, Ollama URL, timeout) in SQLite at `instance/demo_logs.db`. These **take priority over environment variables**. Find the container (Dokploy appends a suffix), then edit:

```bash
docker ps --filter name=ai-guardrails --format '{{.Names}}'

# set the Ollama URL (by container name — Ollama shares dokploy-network)
docker exec <ai-guardrails> python3 -c "
import sqlite3; db=sqlite3.connect('/app/instance/demo_logs.db')
db.execute(\"UPDATE settings SET value='http://ollama-cpu:11434' WHERE key='OLLAMA_API_URL'\"); db.commit()"
docker restart <ai-guardrails>

# raise the timeout (DeepSeek-R1 8B and similar need >= 300s)
docker exec <ai-guardrails> python3 -c "
import sqlite3; db=sqlite3.connect('/app/instance/demo_logs.db')
db.execute(\"UPDATE settings SET value='300' WHERE key='OLLAMA_TIMEOUT'\"); db.commit()"
```

The AI Guardrails playground (`guardrails.<DOMAIN>`) includes a scan-progress modal (minimizable, cancelable, with a local-model warning) and a log-details modal with syntax-highlighted request/response JSON and copy buttons.

## AI Guardrails can't reach Ollama

AI Guardrails reaches Ollama by name (`http://ollama-cpu:11434`) because `ollama-cpu` is on the shared `dokploy-network`. If the playground dropdown shows **"No connection"**, the `/api/tags` probe failed. Check what URL AI Guardrails is actually using — the **settings DB overrides the env**, so a stale DB value (e.g. an old `<SERVER_IP>:11434`) survives redeploys and is the usual culprit:

```bash
docker exec <ai-guardrails> python3 -c "
import sqlite3; c=sqlite3.connect('/app/instance/demo_logs.db').cursor()
c.execute(\"SELECT key,value FROM settings WHERE key='OLLAMA_API_URL'\"); print(c.fetchall())"
# then prove reachability from inside the container:
docker exec <ai-guardrails> sh -lc 'curl -s -m5 http://ollama-cpu:11434/api/tags | head -c 200; echo'
```

If the DB holds a stale value, reset it with the update in [AI Guardrails settings database](#ai-guardrails-settings-database). If the `curl` fails, confirm Ollama is attached to `dokploy-network`: `docker network inspect dokploy-network | grep ollama-cpu` — reattach with `docker network connect dokploy-network ollama-cpu` (the playground compose does this automatically on deploy).

## Traefik file provider (non-Dokploy containers)

For containers not managed by Dokploy's git integration (e.g. a standalone Demo Server), drop a YAML file in `/etc/dokploy/traefik/dynamic/` — Traefik picks it up with no restart:

```yaml
# /etc/dokploy/traefik/dynamic/demo-server.yml
http:
  routers:
    demo-server-web:
      rule: Host(`threat.<DOMAIN>`)
      entryPoints: [web]
      middlewares: [redirect-to-https@file]
      service: demo-server
    demo-server-websecure:
      rule: Host(`threat.<DOMAIN>`)
      entryPoints: [websecure]
      service: demo-server
      tls:
        certResolver: letsencrypt
  services:
    demo-server:
      loadBalancer:
        servers:
          - url: http://<container_name>:8080
        passHostHeader: true
```

The container must also join `dokploy-network` so Traefik can reach it:

```bash
docker network connect dokploy-network <container_name>
```
